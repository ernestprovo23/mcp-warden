# mcp-warden examples

Copy-paste integration examples for the mcp-warden integrity gate. Every file
here is exercised by the `examples` job in
[`.github/workflows/examples.yml`](../.github/workflows/examples.yml): the
workflow YAML is lint-validated and the committed locks under `pinned-servers/`
are re-checked on every run so these examples stay green.

## Pinned-server examples

Each directory under `pinned-servers/` holds a real, openly-available MCP server
(no paid credentials), pinned to a committed `warden.lock`. The locks were
generated by actually running `mcp-warden pin` against each server; the
`examples` CI job re-runs `mcp-warden check` against them so they can never go
stale silently.

| Server | Launcher | Pinned ref | Surface |
|--------|----------|------------|---------|
| [`server-everything`](pinned-servers/server-everything/) | `npx` | `@2026.1.26` | 13 tools, 7 resources, 4 prompts |
| [`server-memory`](pinned-servers/server-memory/) | `npx` | `@2026.1.26` | 9 tools |
| [`server-sequential-thinking`](pinned-servers/server-sequential-thinking/) | `npx` | `@2025.12.18` | 1 tool |

All three are launched via **pinned version refs**, so the supply-chain checks
(`WRD-SUP-*`) do not fire on an unpinned launcher. Each server's surface is also
host-independent (no machine-local schema defaults), so the committed locks
verify identically on macOS and in CI. See each server's `README.md` for the
exact `pin` argv and a sample `check`.

## How the gate works

1. **Pin once** — `mcp-warden pin -- <server launch argv>` captures the declared
   tool/resource/prompt surface into `warden.lock`. Commit the lock.
2. **Check in CI** — `mcp-warden check -- <server launch argv>` re-captures and
   diffs against the lock. Exit 0 = clean, 1 = drift, 2 = capture/IO error.
3. **Re-pin only after review** — when the surface legitimately changes, a human
   reviews the diff and re-pins.
